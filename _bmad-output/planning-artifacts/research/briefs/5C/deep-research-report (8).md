# Walk-Forward Validation (FR29)  
For minute‐frequency FX data (~5.3M bars, 10 years), practitioners typically use multi-year training and multi-month test windows. A common choice is a 2‑year **rolling** training window followed by a 6‑month test window, stepping forward 3 months each time【7†L1745-L1750】. This produces multiple overlapping out‑of‑sample periods. Other options include an **expanding/anchored** window (training always from start date) or a fixed‑length rolling window; the key is to cover different market regimes.  With 10 years of data, one might aim for on the order of 6–12 test segments. As a rule of thumb, at least ~3–5 windows are needed for any significance【36†L1-L4】【7†L1745-L1750】, and more (10–15) windows increase statistical power (at the cost of shorter tests)【7†L1745-L1750】.  

- **Window size:** E.g. train=2 years, test=6 months, step=3 months【7†L1745-L1750】. Adjust so that each test contains enough data (hundreds of trading days) to stabilize Sharpe and drawdown estimates.  
- **Number of windows:** Aim for ~6–12 windows if possible. Fewer than ~3 windows is too few to judge consistency【36†L1-L4】; 10–15 is often cited as a robust target. Too many windows, however, makes each window short and overlapping.  
- **Step size:** Typically overlap windows (e.g. step half the test length) to increase sample count. A 50% overlap (step=3m for 6m test) is common【7†L1745-L1750】. Non‑overlapping steps yield independent results but fewer samples.  
- **Purge/embargo gap:** Leave a buffer between train and test to prevent leakage.  Remove (“purge”) any training samples whose outcome period overlaps the test period – at minimum the strategy’s maximum lookback or holding time. For FX M1 data, this is on the order of hours or days (e.g. ~1–2 days = 1440–2880 bars)【41†L104-L112】. Then apply an **embargo**: skip a further gap after the test (another few hours/days) so that features near the boundary aren’t contaminated【41†L125-L132】. These gaps complement the purging already done inside CV.  
- **Interaction with CV-objective:** Because the optimizer already used K‑fold CV, walk-forward acts as an independent temporal check. We ensure that walk‑forward test periods are always after (in time) their training set. In practice one can simply apply walk-forward on the full data stream (taking care to skip any purged/embargoed regions) – it’s a different slicing than the random folds used in CV【14†L231-L239】. The idea is not to reuse CV folds, but to *additionally* test genuine chronological out-of-sample performance.  
- **Per-window metrics:** Collect full backtest stats for each window: e.g. **Sharpe ratio, CAGR, total return, max drawdown, profit factor, number of trades, win rate**, etc. For example, one walk-forward summary might list per-fold Sharpe, return%, max drawdown%, and trade count【6†L31-L34】.  Consistency is judged by these: large swings or any window with very low/negative Sharpe is a warning.   
- **Aggregation:** Aggregate results by summarizing the distribution of these metrics. Typical summaries include the **mean/median Sharpe** across windows, the **worst-case Sharpe** or drawdown, and the **fraction of windows passing thresholds** (e.g. Sharpe>0). Some practitioners compute a **Walk-Forward Efficiency (WFE)** = (avg OOS Sharpe / in-sample Sharpe)【42†L179-L187】, where values ≳50–60% indicate decent robustness.  More generally, one might require, for example, that at least 80% of windows have Sharpe above a target, or that the worst window Sharpe not fall too far below the average.  In sum, look at both the central tendency and the tails of the per-window metrics; demand no “broken” windows.  Analysis should report all window stats so the operator sees, say, *“Sharpe ranged X–Y, mean Z, with N of M windows >0.5”*.  (See Fig.↓ and sources for illustrations.)【42†L199-L204】【6†L31-L34】  

# CPCV Configuration (FR30)  
Combinatorial Purged CV splits the 10-year series into **N** sequential blocks (groups) and holds out **k** of them as test in each split【14†L236-L244】. For example, with N=10 yearly blocks and k=2, there are C(10,2)=45 possible test combinations (yielding 45 train/test splits). Reasonable choices might be **N≈8–12** and **k=1–3**, so that each test block is several months long but many combinations exist.  If N is large (and C(N,k) is huge), one can sample a random subset of splits to limit runtime【14†L359-L363】.  

- **Group sizing:** With 10 yr data, N=10 (≈1 yr each) or N=12 (10 month each) are typical. k=1 yields 10 folds, k=2 yields 45, etc. Choose k so test sets are not too large; k=1–2 often suffices. The group period should exceed the strategy’s longest indicator/horizon so purge can work effectively.  
- **Purging:** Remove overlapping labels from the end of each training block. In practice, this means discarding any training rows whose “event” (hold period) would intrude into the test segment【41†L104-L112】.  For example, if trades are held up to 1 day, purge ~1 day of bars (1440 bars) from the tail of each training block.  
- **Embargo:** After each test block, leave a time buffer before the next train block. E.g. exclude 1–2 days of bars after each test to avoid feature spillover【41†L125-L132】.  A rule of thumb (from Lopez) is a small percentage of data; practically for M1 we’d use at least several hundred to a few thousand bars.  
- **Independence from CV:** Ensure CPCV fold boundaries don’t coincide with CV test folds. For example, one might align CPCV blocks on fixed calendar periods (years or quarters) distinct from the random folds used earlier. The idea is that CPCV provides *additional* chronology-aware splits on top of the CV robustness already applied.  
- **PBO from CPCV:** Compute the **Probability of Backtest Overfitting (PBO)** using the CPCV results【47†L69-L77】. PBO is the fraction of CPCV splits where the in-sample-best configuration underperforms the median out-of-sample performance. Lower is better. For instance, a PBO of ~0.47 means ~47% of top IS picks failed to beat median OOS【47†L69-L77】. A high PBO (approaching 0.5) implies the strategy is no better than random. We may set rules like: *PBO<0.2 = GREEN, PBO>0.5 = RED, intermediate = YELLOW*. Some practitioners cite PBO<0.2 as a reliability threshold【62†L275-L282】.  

# Parameter Perturbation / Stability (FR31)  
Test robustness by slightly perturbing the optimized parameters and observing performance changes. A practical method is to generate many slightly altered parameter sets around the optimum: e.g. randomly vary each parameter by ±5–10% (or ±1σ of its range). One can do a local random sampling (e.g. 100–500 perturbations for a 15–30 parameter model) or a small multi‑dimensional grid. For each perturbed set, re-run the backtest (or at least compute its key metrics) to create a distribution of outcomes.  

- **Perturbation range:** Use a small fraction of each parameter’s range (e.g. ±10%). Alternatively, fix a radius in parameter space (an L2 ball) around the optimum. The goal is to stay in the “neighborhood” of the solution.  
- **Number of perturbations:** Depends on dimension and compute budget. For ~20 parameters, a few hundred random draws can reveal gross instability; 1000+ yields finer detail. (Note: more is better but check feasibility.)  
- **Stability metrics:** Compare metrics (Sharpe, profit, etc.) across perturbations. Compute, for example, the *max drop* in Sharpe from the optimum, or the standard deviation of Sharpe. One could define a **stability score** = 1 – (max drop / baseline Sharpe). A low score (large drop) means fragility.  
- **Thresholds:** Define pass/fail rules. For instance, if a 10% parameter shift causes >50% drop in Sharpe, that is a RED flag. If all ±10% shifts stay within, say, ±20% of original Sharpe, the strategy is considered stable. In other words, require a “flat” performance surface: robust strategies retain much of their edge under small tweaks.  
- **Visualization:** Plot heatmaps or surface slices for key parameter pairs, or use sensitivity charts. At a minimum, report a scatter or histogram of perturbed Sharpe ratios. For large parameter spaces, one can project into one or two dimensions or report summary statistics. The operator should see, e.g., “Sharpe fell to X% of base under 10% perturbations.”  

# Monte Carlo Simulations (FR32)  
Use synthetic resampling to assess statistical significance and stress resilience.

- **Bootstrap (IID resampling):** Randomly shuffle or resample individual trades (with replacement) to create many alternate equity curves. Run **thousands of simulations** (e.g. 1000–10000). From the simulated distribution, measure the range of outcomes: especially the **5th percentile Sharpe or return** (the “bad luck” case) and the probability of ruin. For instance, if the 5th percentile Sharpe is below 0, that suggests the strategy would often lose out-of-sample. We might require the 5th percentile Sharpe >0 (or >some threshold) for a GREEN pass. Plotting 10,000 MC paths typically yields an equity “cone”: the 50% median path and the 5% “bad‐luck” line【54†L144-L153】. Falling below the 5% line in live trading would be “unlikely luck” (a kill-switch condition). In other words, ensure the live equity stays above the simulated 5th‑percentile envelope.  
- **Permutation (trade-shuffle):** Shuffle the order of trades (preserving individual trade P/L distribution) and recompute performance. Again do ~1000–5000 permutations. Count how often the shuffled Sharpe (or profit) exceeds the original. This gives a non-parametric *p*-value. If >5% of permutations achieve Sharpe ≥ original, the original strategy isn’t statistically significant (chance). Practically, if the permutation *p*-value ≥0.05, flag a warning. Permutation testing isolates “sequence risk”: if the strategy relies on fortunate trade ordering or autocorrelation, the original will look unusually good. Note: permutation preserves profit factor and tests drawdown variability【54†L79-L88】.  
- **Stress testing (costs, market changes):** Increase trading costs (spread, slippage) to see how performance degrades. For example, multiply all spreads/slippage by 1.5×, 2×, 3×. If the strategy still has positive expectation at 1.5× (GREEN) but only breaks even at 2× (YELLOW), that’s better than failing already at 1.5× (RED). One can similarly stress-test latency or slippage patterns. Also consider adverse market conditions: e.g. simulate scenarios with less favorable regime mix (fewer “good” days).  For FX M1 specifically, stress spreads and slippage by at least +50–100%. As an illustration, [62] recommends adding ~10–20% to the bid/ask spread as a noise test【62†L319-L323】. If performance collapses under moderate stress, the strategy is not robust.  
- **Interpretation:** Look at, e.g., the probability of blow-up or max drawdown distribution. A common criterion is *“the strategy must survive the 5th percentile scenario”*【54†L140-L149】. Also ensure metrics like Sharpe or profit remain within acceptable bounds under stress. If, say, Sharpe drops from 1.5 to 0.5 at 1.5× costs, that would lower the confidence score.  

# Regime Analysis (FR33)  
Partition the data into market regimes (typically by volatility) and check performance in each. A standard approach is to compute a realized-volatility measure (e.g. 20-day ATR or stddev) and split into low/medium/high (terciles) regimes. Regime can also incorporate session timing (Asian/London/NY) if desired, but beware too many buckets: 3 vol regimes × 3 sessions = 9 buckets. Only do session splits if each bucket still has enough trades. 

- **Regime definition:** For example, compute 20-day rolling volatility (or ATR) and define low/med/high as the bottom/middle/top 1/3 of values. Alternatively, use a Hidden Markov Model on returns to infer regimes (more complex). Terciles or quartiles are simpler and transparent.  
- **Session interaction:** If splitting by session (3 daily FX sessions), ensure each session-regime bucket has a minimum number of trades. As a rule of thumb, aim for ≥30–50 trades per bucket for statistical meaning. If many buckets have very few trades, collapse some buckets or skip the dimension. The analysis only makes sense where sample size allows.  
- **Per-regime metrics:** Compute the same metrics (Sharpe, return, drawdown) within each regime (and session, if used). Identify any regime where performance is markedly worse. For instance, if “high-volatility” regime yields negative Sharpe or net loss, that’s a red flag.  
- **Failure criteria:** We may treat extreme underperformance in any regime as a serious issue. E.g. if one regime has Sharpe<0 after costs, that could warrant at least a YELLOW (maybe RED if it is a dominant regime). However, if the strategy fails only in a rare regime but performs strongly in the others, one might tolerate YELLOW (with note). The operator should see a table of regime performance. As an example rule: *if any regime loses money on average, mark as YELLOW; if two or more regimes lose, mark RED*. (Exact cutoffs depend on risk tolerance.)  

# Confidence Score Aggregation (FR34)  
To form an overall RED/YELLOW/GREEN rating, combine all test results via a hybrid gate+score approach. Some checks should be **hard gates** (fail immediately), others can feed into a composite score. For example:  

- **Hard gates:** If any critical test fails, assign RED outright. Such gates might include: negative cumulative return in *any* walk-forward window, PBO > 0.5, >10% Sharpe drop under 10% parameter shift, 5th-percentile MC Sharpe <0, or strategy losing money in multiple regimes. Failing a hard gate signals obvious overfitting or instability.  
- **Soft scores:** For the remaining tests, convert results into normalized scores (0–1) and weight them. Example components: average OOS Sharpe (normalized to a target, e.g. Sharpe=1→1.0), stability score (1 – relative Sharpe drop), (1 – PBO), MC 5% Sharpe relative to baseline, worst-regime Sharpe relative to overall.  For instance: 
  - **Sharpe score:** = min(Sharpe_avg/2, 1) (cap at 1)  
  - **PBO score:** = 1 – PBO (so PBO=0→1, PBO=0.5→0.5)  
  - **Stability score:** = 1 – (max perturbation drop fraction)  
  - **Monte Carlo score:** = (Sharpe_5th_percentile / target_Sharpe) clipped [0,1].  
  - **Regime score:** = min(Regrime_best/target, 1) where “best” might be worst-case Sharpe.  
  Then compute a weighted average: e.g. 30% Sharpe, 20% PBO, 20% stability, 20% MC, 10% regime.  

- **Thresholds:** Finally, map the combined score to R/Y/G. For example: score ≥0.8 = GREEN; 0.5–0.8 = YELLOW; <0.5 = RED.  Alternatively, require all components above sub-thresholds (e.g. Sharpe>1, PBO<0.3, stability>0.8) for GREEN, and assign YELLOW/RED otherwise. The exact formula can be tuned on known good vs bad strategies.  One guideline: Sophie’s framework recommends PBO<0.2 and deflated Sharpe>1 as criteria for green【62†L275-L282】.  

- **Breakdown display:** Present a **scorecard** showing each component. For instance, a bar chart or table listing: “OOS Sharpe: X (score Y%)”, “PBO: Z (score Y%)”, “MC worst-case Sharpe: X (score Y%)”, etc. This helps the reviewer see which tests drive the result. A visual “pie” or stacked bar of weighted scores can also clarify the contributions.  

*(There is no single published “confidence score” formula; this is a design choice. The key is that the combination be intuitive and reproducible.)*  

# Validation Gauntlet Ordering and Short-Circuiting  
Arrange the tests roughly from cheapest to most expensive, and stop early on catastrophic failure. A suggested order is:  

1. **Quick sanity checks / parameter jitter:** Verify the optimized strategy has reasonable raw stats (Sharpe>0, trades>min) and test a few perturbations. If it fails basic logic or is wildly unstable, reject early.  
2. **Walk-forward analysis:** Run the full WFA. If Sharpe collapses or most windows fail, mark RED and skip further heavy tests.  
3. **Monte Carlo (bootstrap & permutation):** Simulate random resamplings. If the strategy is already in doubt, these may confirm failure.  
4. **Stress tests (cost & slippage):** Apply widened spreads/slippage. If performance is acceptable, proceed; if the strategy dies under modest stress, mark RED or YELLOW.  
5. **Regime analysis:** Evaluate per-regime performance. A single failing regime can downgrade to YELLOW.  
6. **CPCV/PBO check (final):** Finally, run the combinatorial CV/PBO. This is computationally heavy, so do it last. If everything else passes but PBO is high, then mark RED at the end【62†L269-L277】.  

This ordering (cheapest filters first) conserves time. In practice, one might also interleave or parallelize some tests. The key is to *short-circuit*: e.g. if walk-forward already shows zero trading (or negative returns) in all windows, there’s no need to run CPCV or MC. Likewise, if CPCV yields PBO>0.5, you can stop there (RED) without further analysis. The final decision should always consider all results, but failing any critical gate should immediately yield a RED rating.
