## 1. `@fix_plan.md` Format

The **generated** `@fix_plan.md` (produced by `bmalph implement`) has this exact structure:

```markdown
# Ralph Fix Plan

## Stories to Implement

### Epic Name Here
> Goal: Epic description text

- [ ] Story 1.1: Story Title Here
  > As a user, I want X
  > So that Y
  > AC: Given condition, When action, Then result
  > AC: Given condition2, When action2, Then result2
  > Spec: specs/planning-artifacts/stories.md#story-1-1

- [ ] Story 1.2: Another Story Title
  > Description line
  > AC: acceptance criterion
  > Spec: specs/planning-artifacts/stories.md#story-1-2

### Another Epic
> Goal: Another epic description

- [ ] Story 2.1: ...

## Completed

## Notes
- Follow TDD methodology (red-green-refactor)
- One story per Ralph loop iteration
- Update this file after completing each story
```

Key details:
- Stories are grouped under `### Epic Name` headings with `> Goal:` descriptions
- Each story is a checkbox: `- [ ] Story {id}: {title}`
- Description lines use `> ` prefix (max 3 lines, split on sentence boundaries)
- Acceptance criteria use `> AC:` prefix
- Spec links point to `specs/planning-artifacts/{filename}#story-{anchor}`
- Ralph marks stories `[x]` when complete; progress is preserved across re-runs

The **template** `fix_plan.md` (in `ralph/templates/`) is a generic placeholder with High/Medium/Low priority sections -- it is NOT used by `bmalph implement`. The `generateFixPlan()` function in `src/transition/fix-plan.ts` builds the real one from parsed stories.

## 3. What `bmalph implement` Does (the transition)

Source: `src/commands/implement.ts` -> `src/transition/orchestration.ts`

Step-by-step:

1. **Checks re-run protection**: If `.ralph/@fix_plan.md` already exists, warns and exits unless `--force` is passed. Progress (checked items) is preserved on re-run.

2. **Locates BMAD artifacts** (`artifacts.ts`): Searches these paths in order:
   - `_bmad-output/planning-artifacts`
   - `_bmad-output/planning_artifacts`
   - `docs/planning`

3. **Collects artifacts** (`artifact-collection.ts`): Classifies files by regex:
   - PRD files: filename contains `prd`
   - Architecture files: filename contains `architect`
   - Story files: filename contains `epic(s)` or `story/stories`
   - Readiness files: filename contains `readiness`

4. **Parses stories** (`story-parsing.ts`): Reads epics/stories markdown expecting:
   - Epic headers: `## Epic N: Title`
   - Story headers: `### Story N.M: Title` (regex: `STORY_HEADER_PATTERN`)
   - Description: lines between story header and acceptance criteria
   - Acceptance criteria: Given/When/Then format blocks

5. **Syncs fix plan** (`fix-plan-sync.ts`): Generates `@fix_plan.md` from parsed stories. If existing plan has checked items, merges progress (title-based matching or sprint-status.yaml).

6. **Prepares specs directory**: Copies all BMAD artifacts to `.ralph/specs/` with changelog tracking.

7. **Generates context outputs** (`context-output.ts`):
   - `PROJECT_CONTEXT.md` -- extracted from PRD (goals, scope, NFRs) and architecture
   - `PROMPT.md` -- replaces `[YOUR PROJECT NAME]` placeholder with project name, or regenerates from template
   - `@AGENT.md` -- customized with detected tech stack
   - `SPECS_INDEX.md` -- index of spec files

8. **Updates state**: Sets phase to 4 ("implementing") in `bmalph/state/`.

## 4. BMAD Output Format bmalph Expects

bmalph reads from `_bmad-output/planning-artifacts/` (primary) or `docs/planning/` (fallback).

**Required artifacts** (will error if missing):
- PRD (`*prd*.md`)
- Architecture (`*architect*.md`)
- Epics & Stories (`*epic*.md` or `*stor*.md`)
- Readiness Report (`*readiness*.md`)

**Optional artifacts:**
- Product Brief (`*brief*.md`)
- Market Research (`*market*.md`)
- Domain Research (`*domain*.md`)
- Technical Research (`*tech*research*.md`)
- UX Design (`*ux*.md`)
- Sprint Status (`sprint-status.yaml` or `.yml`)

**Story file format expected:**

```markdown
## Epic 1: Epic Title Here

Optional epic description lines here.

### Story 1.1: Story Title

As a [role], I want [feature], So that [benefit].

**Acceptance Criteria:**

- **Given** some precondition
  **When** some action
  **Then** some result

- **Given** another precondition
  **When** another action
  **Then** another result
```

Story IDs must follow the `N.M` pattern (e.g., `1.1`, `2.3`). Non-canonical IDs generate warnings.

## 5. Does `bmalph init` Overwrite Existing Installations?

**Key behaviors:**

- **If already initialized** (`bmalph/config.json` exists): Refuses to run, tells you to use `bmalph upgrade` instead.
- **If `_bmad/` exists** (standalone BMAD): Prints "Existing BMAD installation detected", then REPLACES `_bmad/` framework files with the bmalph-managed version.
- **`_bmad-output/` is NEVER touched** -- your planning artifacts (PRDs, architecture, stories) are safe.
- **`.claude/` directory**: bmalph ADDS `.claude/commands/` slash commands for Claude Code platform. It does NOT delete existing `.claude/` config or settings files.
- **Instructions files** (e.g., `CLAUDE.md`): Uses a MERGE strategy (`mergeInstructionsFile`) -- it adds bmalph-managed sections rather than overwriting the whole file.

**Bottom line**: `bmalph init` replaces `_bmad/` framework files and adds its own sections to instructions files, but preserves `_bmad-output/`, `.claude/settings.json`, and existing instructions content.

## 7. Using Ralph Standalone Without bmalph's BMAD Copy

**Partially possible, but not a supported path:**

- Ralph's loop scripts (`ralph_loop.sh`, drivers) are self-contained shell scripts that read `.ralph/PROMPT.md`, `.ralph/@fix_plan.md`, and `.ralph/@AGENT.md`.
- The `.ralphrc` config supports a legacy project-root `.ralphrc` fallback for "older standalone Ralph layouts."
- However, `bmalph implement` is the only supported way to generate `@fix_plan.md` from BMAD stories. Without it, you would need to manually create the fix plan in the expected checkbox format.
- The PROMPT.md template uses `[YOUR PROJECT NAME]` placeholder -- you could manually fill it.
- You could theoretically copy `ralph/` to `.ralph/`, manually create `@fix_plan.md` and `PROMPT.md`, and run `bash .ralph/ralph_loop.sh` directly.
- The README references a legacy `frankbria/ralph-claude-code` repo in the ralphrc template comments, suggesting Ralph existed as a standalone project before bmalph integration.

**Practical recommendation**: Use `bmalph init` + `bmalph implement` for the supported workflow. Manual standalone setup is possible but unsupported and fragile.
