- The documented operator workflow is not runnable as written. The skills `cd` to repo root and call `python -m strategy ...` ([`strategy_review.md:21`](\/c\/Users\/ROG\/Projects\/Forex%20Pipeline\/.claude\/skills\/strategy_review.md#L21), [`strategy_review.md:27`](\/c\/Users\/ROG\/Projects\/Forex%20Pipeline\/.claude\/skills\/strategy_review.md#L27), [`strategy_update.md:26`](\/c\/Users\/ROG\/Projects\/Forex%20Pipeline\/.claude\/skills\/strategy_update.md#L26)), and the CLI usage string matches that ([`__main__.py:4`](\/c\/Users\/ROG\/Projects\/Forex%20Pipeline\/src\/python\/strategy\/__main__.py#L4)). From repo root, `python -m strategy` will not resolve without extra path setup, and the story-specified `python -m src.python.strategy ...` path was not implemented.
- Version overflow past `v999` breaks immutability and latest-version lookup. Story 2.5 explicitly requires `v999 -> v1000`, but shared storage still matches only exactly three digits ([`storage.py:23`](\/c\/Users\/ROG\/Projects\/Forex%20Pipeline\/src\/python\/strategy\/storage.py#L23), [`storage.py:114`](\/c\/Users\/ROG\/Projects\/Forex%20Pipeline\/src\/python\/strategy\/storage.py#L114)). `save_strategy_spec()` derives the next version from `list_versions()` ([`storage.py:46`](\/c\/Users\/ROG\/Projects\/Forex%20Pipeline\/src\/python\/strategy\/storage.py#L46)), so once `v1000.toml` exists it is ignored and the next save targets `v1000.toml` again, silently overwriting it. The review CLI also depends on `load_latest_version()` for default review selection ([`__main__.py:33`](\/c\/Users\/ROG\/Projects\/Forex%20Pipeline\/src\/python\/strategy\/__main__.py#L33), [`__main__.py:43`](\/c\/Users\/ROG\/Projects\/Forex%20Pipeline\/src\/python\/strategy\/__main__.py#L43)).
- Diff output is not reliably “plain English” for common modification types, so AC5 is only partially met. Generic add/remove/change descriptions are built from field paths and `_format_value()` output ([`versioner.py:134`](\/c\/Users\/ROG\/Projects\/Forex%20Pipeline\/src\/python\/strategy\/versioner.py#L134), [`versioner.py:170`](\/c\/Users\/ROG\/Projects\/Forex%20Pipeline\/src\/python\/strategy\/versioner.py#L170), [`versioner.py:177`](\/c\/Users\/ROG\/Projects\/Forex%20Pipeline\/src\/python\/strategy\/versioner.py#L177), [`versioner.py:191`](\/c\/Users\/ROG\/Projects\/Forex%20Pipeline\/src\/python\/strategy\/versioner.py#L191), [`versioner.py:242`](\/c\/Users\/ROG\/Projects\/Forex%20Pipeline\/src\/python\/strategy\/versioner.py#L242)). **MEDIUM findings**
- `update_manifest_version()` always sets `current_version` to the entry being updated ([`versioner.py:408`](\/c\/Users\/ROG\/Projects\/Forex%20Pipeline\/src\/python\/strategy\/versioner.py#L408), [`versioner.py:440`](\/c\/Users\/ROG\/Projects\/Forex%20Pipeline\/src\/python\/strategy\/versioner.py#L440)). If an older version is later confirmed or edited in-place, `current_version` can regress away from the actual highest saved artifact.
- Modification path validation is too permissive. `_is_valid_field_path()` accepts any subpath under a whitelisted prefix ([`modifier.py:134`](\/c\/Users\/ROG\/Projects\/Forex%20Pipeline\/src\/python\/strategy\/modifier.py#L134), [`modifier.py:141`](\/c\/Users\/ROG\/Projects\/Forex%20Pipeline\/src\/python\/strategy\/modifier.py#L141)), but `_set_nested()` / `_add_to_list()` / `_remove_from_list()` then blindly index into the structure ([`modifier.py:194`](\/c\/Users\/ROG\/Projects\/Forex%20Pipeline\/src\/python\/strategy\/modifier.py#L194), [`modifier.py:211`](\/c\/Users\/ROG\/Projects\/Forex%20Pipeline\/src\/python\/strategy\/modifier.py#L211), [`modifier.py:226`](\/c\/Users\/ROG\/Projects\/Forex%20Pipeline\/src\/python\/strategy\/modifier.py#L226)). Invalid inputs will surface as raw `TypeError`/`IndexError` rather than a controlled validation error.
- The code comments claim `spec_hash` is a content hash excluding lifecycle metadata, but the actual hasher serializes the full model dump except `_`-prefixed keys ([`confirmer.py:173`](\/c\/Users\/ROG\/Projects\/Forex%20Pipeline\/src\/python\/strategy\/confirmer.py#L173), [`hasher.py:28`](\/c\/Users\/ROG\/Projects\/Forex%20Pipeline\/src\/python\/strategy\/hasher.py#L28)). That means status/config-hash changes alter `spec_hash`, weakening it as stable content identity.
