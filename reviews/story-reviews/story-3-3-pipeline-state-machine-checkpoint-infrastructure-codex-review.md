## Recommended Changes
1. Move the implementation target from `src/python/pipeline/` to the architecture-approved `src/python/orchestrator/`, or explicitly update the architecture first.
2. Replace `operator_approved: bool` with an explicit gate decision model supporting `accept`, `reject`, and `refine`, plus reason and timestamp.
3. Expand `PipelineState` and `get_status()` to include `gate_status`, `blocking_reason`, `decision_required`, `last_outcome`, `manifest_ref`, and `evidence_pack_ref`.
4. Stop defining shared contracts as ad hoc `dict` fields; add schema-backed types for stage results, checkpoints, errors, and status payloads via `contracts/`.
5. Treat Story 3.2 contract consumption as mandatory, not optional fallback; fix the broken dependency path in the story file.
6. Narrow 3.3 to cross-stage resume and state persistence; make within-stage checkpoint recovery contract-driven and integrated with Story 3.4.
7. Remove hardcoded retry counts; use D7 config values for retry attempts/backoff and keep alerting aligned with the monitoring/alerting layer.
8. Resolve the `clean_partial_files()` versus `partial_artifact_path` conflict by defining exact cleanup order and what files are safe to delete.
9. Replace the fixed single-successor stage graph with an extensible stage definition model, or at minimum document how future stages are added without schema rewrite.
10. Add tests for reject/refine paths, status “why” fields, run/version lineage across reruns, config-hash mismatch on resume, and contract conformance against 3.2 outputs.
