## 3. Architecture Challenge
- **Assessment:** CONCERN
- **Evidence:** D1, D13, and D14 are sound for this story: subprocess isolation, cost model as a library, and shared evaluation logic are all architecturally aligned ([architecture.md](/c/Users/ROG/Projects/Forex Pipeline/_bmad-output/planning-artifacts/architecture.md#L314), [architecture.md](/c/Users/ROG/Projects/Forex Pipeline/_bmad-output/planning-artifacts/architecture.md#L901), [architecture.md](/c/Users/ROG/Projects/Forex Pipeline/_bmad-output/planning-artifacts/architecture.md#L937)). The problem is the story violates that separation in places.
- **Observations:** The biggest conflict is exit logic ownership. Architecture says `strategy_engine` includes exit rule evaluation, including `exits.rs` ([architecture.md](/c/Users/ROG/Projects/Forex Pipeline/_bmad-output/planning-artifacts/architecture.md#L937), [architecture.md](/c/Users/ROG/Projects/Forex Pipeline/_bmad-output/planning-artifacts/architecture.md#L1660)), but the story also implements stop loss, take profit, trailing stop, and chandelier exit inside `position.rs` ([story 3.5](/c/Users/ROG/Projects/Forex Pipeline/_bmad-output/implementation-artifacts/3-5-rust-backtester-crate-trade-simulation-engine.md#L80)). That is a direct duplication risk. The determinism contract is also internally inconsistent: AC #8 says bit-identical outputs, but the tests explicitly warn Arrow IPC may vary at the byte level, and `run_metadata.json` includes a timestamp ([story 3.5](/c/Users/ROG/Projects/Forex Pipeline/_bmad-output/implementation-artifacts/3-5-rust-backtester-crate-trade-simulation-engine.md#L41), [story 3.5](/c/Users/ROG/Projects/Forex Pipeline/_bmad-output/implementation-artifacts/3-5-rust-backtester-crate-trade-simulation-engine.md#L140), [story 3.5](/c/Users/ROG/Projects/Forex Pipeline/_bmad-output/implementation-artifacts/3-5-rust-backtester-crate-trade-simulation-engine.md#L164)).
- **Recommendation:** Make architecture ownership explicit in the story. `strategy_engine` should own signal and exit decisions; backtester should own state transitions, fills, cost application, and artifact emission. Redefine determinism as either canonical-value equality or canonical-byte equality, and keep non-deterministic metadata out of that contract.

## 4. Story Design
- **Assessment:** CRITICAL
- **Evidence:** The acceptance criteria look testable on paper, but key semantics are underspecified or contradictory. AC #2 says evaluate entry, then exit, then position management ([story 3.5](/c/Users/ROG/Projects/Forex Pipeline/_bmad-output/implementation-artifacts/3-5-rust-backtester-crate-trade-simulation-engine.md#L17)), while the main loop opens positions before exit checks in the same pass ([story 3.5](/c/Users/ROG/Projects/Forex Pipeline/_bmad-output/implementation-artifacts/3-5-rust-backtester-crate-trade-simulation-engine.md#L106)). Task 5 says fills occur on the signal bar, with next-bar-open “if configured,” which leaves live-parity timing ambiguous ([story 3.5](/c/Users/ROG/Projects/Forex Pipeline/_bmad-output/implementation-artifacts/3-5-rust-backtester-crate-trade-simulation-engine.md#L97)).
- **Observations:** Several real gaps remain. There is no explicit policy for what happens to an open position during quarantined bars; “skip the bar” can silently suppress exits and distort drawdown. The trade schema stores one `session`, but trades can span sessions, and later analysis wants session breakdowns. The equity curve is defined as closed-trade cumulative P&L, which may understate drawdown and distort downstream evidence packs. The memory model relies on heuristics like “1 trade per 100 bars,” which is not a defensible deterministic capacity plan.
- **Recommendation:** Rewrite the ACs around explicit contracts: bar timing, fill timing, quarantine handling, open-position policy, equity definition, session attribution, and determinism scope. Without that, implementation will pass tests while still producing system-level wrongness.

- **Observations:** The current output contract is probably too thin for downstream fidelity work. The story persists cost-adjusted prices but not the raw price, spread applied, slippage applied, or separate entry/exit sessions. That means later reconciliation and operator evidence may lack the provenance needed to explain divergence without re-simulating internals. This is exactly the kind of omission that causes a rewrite in later epics.
- **Recommendation:** Treat Story 3.5 outputs as canonical forensic artifacts, not just backtest summaries. Persist enough attribution now to support 3.6, 3.7, and later reconciliation without reopening the core engine contract.

## Recommended Changes
1. Rewrite the story boundary so 3.5 explicitly owns simulation correctness and canonical result emission, while 3.3/3.4/3.6 own orchestration, bridge behavior, and versioned artifact management.
2. Replace the current determinism language with a precise contract: “canonical data values are identical” or “serialized Arrow bytes are identical,” and explicitly exclude `progress.json`, checkpoints, logs, and timestamped metadata if needed.
3. Remove or clarify `run_metadata.json` timestamp behavior so it does not contradict the determinism claim.
4. Define one fill-timing model for V1 that matches MT5-supported behavior; do not leave same-bar vs next-bar-open as an implicit configurable.
5. Resolve exit-logic ownership: either `strategy_engine` emits exit decisions, or `backtester` does, but not both.
6. Add an explicit quarantine policy for open positions during quarantined spans.
7. Expand canonical trade outputs to include raw fill prices, spread/slippage applied, separate entry and exit sessions, and signal identifiers.
8. Fix the metric contract: include max drawdown amount if the AC requires it, and either define the capital base for `return_pct` or remove it.
9. Replace the proposed Sharpe calculation with a documented method aligned to the system’s timeframe and equity definition.
10. Replace heuristic memory sizing like “1 trade per 100 bars” with a bounded streaming design plus startup preflight checks.
11. Decide whether the equity curve is realized-only or mark-to-market; if realized-only, say so explicitly, and consider emitting both.
12. Add a short downstream contract section stating exactly what 3.6 and 3.7 may rely on from this story’s outputs.
